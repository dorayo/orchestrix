# Orchestrix 版本发布指南

## 🚀 自动化版本管理系统

Orchestrix 采用完全自动化的版本管理系统，基于 semantic-release 和 conventional commits 实现零人工干预的版本发布流程。

---

## 📋 版本管理架构

### 分层版本管理策略

```
Layer 1: 主版本自动化管理 (Semantic Release)
  ├── package.json                    # 主项目版本
  ├── package-lock.json              # 锁定文件自动同步
  ├── tools/installer/package.json   # 安装器版本自动同步
  ├── orchestrix-core/core-config.yaml  # 核心配置自动同步
  └── tools/cli.js                    # 动态版本读取

Layer 2: 扩展包独立管理
  └── expansion-packs/*/config.yaml   # 各扩展包独立版本
```

### 自动同步机制

所有版本相关文件通过 `semantic-release-sync-all.js` 插件自动同步：

- ✅ **自动修复** CLI 硬编码版本问题
- ✅ **智能同步** 所有关联文件
- ✅ **完整日志** 详细的版本同步记录
- ✅ **错误处理** 优雅的异常处理机制

## 🎯 发布工作流程

### 主版本发布（完全自动化）

#### 1. 使用 Conventional Commits

```bash
# 功能特性 (minor 版本升级)
git commit -m "feat: add new version management system"

# 修复问题 (patch 版本升级)
git commit -m "fix: resolve CLI version display issue"

# 破坏性变更 (major 版本升级)
git commit -m "feat!: redesign version management architecture"

# 或使用详细格式
git commit -m "feat: add automated release system

- Implement semantic-release workflow
- Add unified version synchronization
- Remove redundant version scripts

BREAKING CHANGE: Manual version scripts are removed"
```

#### 2. 自动发布流程

```bash
# 方式一：推送到主分支自动触发
git push origin main

# 方式二：手动触发发布
npm run release

# 方式三：测试发布流程（dry-run）
npm run release:test
```

#### 3. 发布后自动操作

- 📦 自动生成 CHANGELOG.md
- 🏷️ 创建 Git 标签
- 📝 生成 GitHub Release
- 🚀 发布到 npm 仓库
- 🔄 同步所有版本相关文件

## 🧩 扩展包版本管理

### 单个扩展包操作

```bash
# 语义化版本升级
npm run expansion:bump <pack-name> [major|minor|patch]

# 示例：升级游戏开发扩展包
npm run expansion:bump orchestrix-2d-phaser-game-dev minor

# 设置特定版本
npm run expansion:set <pack-name> <version>

# 示例：设置创建工具包版本
npm run expansion:set orchestrix-creator-tools 2.1.0
```

### 批量扩展包操作

```bash
# 批量语义化升级
npm run expansion:bump-all [major|minor|patch]

# 示例命令
npm run expansion:bump-all patch    # 所有扩展包 patch 版本 +1
npm run expansion:bump-all minor    # 所有扩展包 minor 版本 +1
npm run expansion:bump-all major    # 所有扩展包 major 版本 +1
```

## 📊 版本号规范

### Semantic Versioning

遵循 [SemVer 2.0.0](https://semver.org/) 规范：

```
主版本.次版本.修订版本 (MAJOR.MINOR.PATCH)

示例：3.1.1
├── 3: 主版本（重大变更）
├── 1: 次版本（功能增加）
└── 1: 修订版本（问题修复）
```

### Commit 类型映射

| Commit 类型                    | 版本影响 | 说明               |
| ------------------------------ | -------- | ------------------ |
| `feat:`                        | minor    | 新功能             |
| `fix:`                         | patch    | 问题修复           |
| `feat!:` 或 `BREAKING CHANGE:` | major    | 破坏性变更         |
| `docs:`                        | 无       | 文档更新           |
| `chore:`                       | 无       | 构建/工具变更      |
| `refactor:`                    | 无       | 重构（无功能变更） |
| `test:`                        | 无       | 测试相关           |

### 版本发布检查清单

**发布前检查：**

- [ ] 所有测试通过
- [ ] 文档更新完成
- [ ] Commit 消息符合规范
- [ ] 工作目录干净（无未提交变更）

**发布后验证：**

- [ ] 版本号正确升级
- [ ] 所有文件版本同步
- [ ] GitHub Release 创建成功
- [ ] npm 包发布成功
- [ ] CHANGELOG 自动生成

## 🔧 故障排除

### 常见问题解决

#### 版本同步失败

```bash
# 检查文件权限
ls -la tools/semantic-release-sync-all.js

# 手动测试同步插件
npm run release:test
```

#### CLI 版本显示错误

```bash
# 确认 CLI 使用动态版本
grep "require('../package.json').version" tools/cli.js

# 如果仍然硬编码，手动修复
sed -i 's/.version('\''[0-9.]*'\'')/.version(require('\''..\/package.json'\'').version)/g' tools/cli.js
```

#### 扩展包版本不一致

```bash
# 检查扩展包配置
ls -la expansion-packs/*/config.yaml

# 重新同步扩展包版本
npm run expansion:bump-all patch
```

## 🎛️ 配置文件说明

### semantic-release 配置

`.releaserc.json` - 发布流程配置：

```json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer", // 分析 commit
    "@semantic-release/release-notes-generator", // 生成发布说明
    "@semantic-release/changelog", // 更新 CHANGELOG
    "@semantic-release/npm", // npm 发布
    "./tools/semantic-release-sync-all.js", // 自定义同步插件
    "@semantic-release/git", // Git 操作
    "@semantic-release/github" // GitHub Release
  ]
}
```

### 自定义同步插件

`tools/semantic-release-sync-all.js` - 核心同步逻辑：

- 🎯 统一版本同步
- 🔧 CLI 动态版本修复
- 📝 详细操作日志
- ⚠️ 优雅错误处理

## 📈 版本历史管理

### 查看版本历史

```bash
# Git 标签历史
git tag --sort=-version:refname | head -10

# 发布历史
npm view orchestrix versions --json

# 本地 CHANGELOG
cat CHANGELOG.md
```

### 版本回滚（紧急情况）

```bash
# 注意：生产环境谨慎使用

# 回滚到指定版本
npm version <previous-version> --no-git-tag-version

# 手动同步所有文件
node tools/semantic-release-sync-all.js  # 需要实现 CLI 接口

# 强制推送
git add -A
git commit -m "chore: emergency rollback to <previous-version>"
git push origin main
```

## 🏆 最佳实践

### ✅ 推荐做法

1. **规范化提交**
   - 使用 conventional commits 格式
   - 清晰描述变更内容
   - 包含必要的 BREAKING CHANGE 说明

2. **版本发布**
   - 优先使用自动发布流程
   - 发布前运行 `npm run release:test` 预览
   - 保持 main 分支稳定

3. **扩展包管理**
   - 扩展包版本保持独立
   - 批量操作前先测试单个包
   - 记录扩展包版本升级原因

### ❌ 避免事项

1. **手动版本管理**
   - ❌ 不要手动修改 package.json 版本
   - ❌ 不要使用已删除的版本脚本
   - ❌ 不要跳过 conventional commit 格式

2. **发布流程**
   - ❌ 不要在不稳定分支发布
   - ❌ 不要忽略测试结果
   - ❌ 不要手动创建 Git 标签
