# 技术偏好一致性传递保证机制

## 核心目标

确保用户在`technical-preferences.md`中定义的技术偏好能够完整、准确地从Architect Agent传递到Dev Agent和QA Agent，在整个开发流程中保持技术决策的一致性执行。

## 技术偏好传递链条完整性

```
用户技术偏好 (technical-preferences.md)
↓ (Architect Agent 解读并架构化)
架构文档体系 (tech-stack.md, coding-standards.md等)
↓ (SM Agent 提取并故事化)
Story Dev Notes - 技术偏好摘要
↓ (Dev Agent 读取并实现)
符合偏好的代码实现
↓ (QA Agent 验证并确认)
技术偏好合规验证
```

## 一致性保证机制体系

### 1. 源头一致性确保 - Story模板规范

**技术偏好摘要强制要求** (`story-tmpl.yaml`)：

- Dev Notes必须包含"Technical Preferences Summary"
- 明确记录适用的技术栈选择
- 详细说明架构模式要求
- 列出开发实践约束
- 指明库/框架使用偏好

**目的**：确保用户技术偏好在story层面得到完整体现，为后续一致性执行奠定基础。

### 2. 传递一致性监控 - Story创建流程

**SM Agent技术偏好提取职责** (`create-next-story.md`)：

- 从架构文档中准确提取技术偏好要求
- 将技术约束转化为具体的开发指导
- 建立technical-preferences.md与story实现的直接映射
- 提供明确的源文档引用链条

**目的**：防止技术偏好在从架构层转向实现层时发生误解或遗漏。

### 3. 执行一致性验证 - 四层检查机制

#### 3.1 规划阶段一致性检查

**PO Master验证** (`po-master-checklist.md`)：

- 验证所有项目文档中技术选择的一致性
- 确保story层面的技术要求与原始偏好对齐
- 检查不同制品间是否存在技术决策冲突

#### 3.2 Story创建阶段严谨性验证

**SM Agent内部质量保证** (`create-next-story.md`, `sm-technical-extraction-checklist.md`)：

- 强制性技术细节提取验证（80%完成率要求）
- 技术架构文档引用准确性检查
- Story技术指导完整性验证（≥7/10分通过）
- 量化质量评分和门控机制

**Story质量专项验证** (`validate-story-quality.md`)：

- 独立的8步骤全面质量验证流程
- 技术准确性、可行性、完整性评估
- 严格的质量评分标准（1-10分制）

#### 3.3 Story技术准确性审核

**Architect Agent技术审核** (`review-story-technical-accuracy.md`)：

- 架构一致性专业验证（第二层质量保证）
- 技术组件对架构文档的准确性验证
- 实现可行性和技术依赖完整性评估
- 10分制技术准确性评分（≥7分通过）
- Critical/Major/Minor三级问题分类

#### 3.4 准备阶段一致性审核

**Story验证流程** (`validate-next-story.md`, `story-draft-checklist.md`)：

- 验证技术偏好信息在story中的完整性
- 检查技术要求的可执行性和明确性
- 确认dev agent能够获得充分的技术指导

#### 3.5 实现阶段一致性监督

**QA合规审查** (`review-story.md`)：

- 验证代码实现是否遵循原始技术偏好
- 检查技术栈使用的合规性
- 确认架构模式的正确执行

## 一致性保证检查点矩阵

| 检查阶段     | 检查主体        | 验证内容             | 一致性目标            | 质量门控           |
| ------------ | --------------- | -------------------- | --------------------- | ------------------ |
| **规划阶段** | PO Agent        | 文档间技术选择一致性 | 架构→Story一致        | 跨文档一致性检查   |
| **故事创建** | SM Agent        | 技术细节提取验证     | Preferences→Story提取 | ≥80%完成率+≥7/10分 |
| **技术审核** | Architect Agent | 技术准确性专业审核   | 架构→Story准确性      | ≥7/10分技术准确性  |
| **实现验证** | QA Agent        | 代码技术合规性       | Story→Code一致        | 技术合规验证       |

## 技术偏好执行质量保证

### 对Architect Agent的一致性要求

- 忠实解读用户technical-preferences.md
- 在架构文档中完整体现技术偏好
- 确保技术选择的逻辑一致性和可执行性

### 对SM Agent的一致性要求

- 准确从架构文档提取技术偏好要求
- 在story创建时完整传递技术约束
- 建立清晰的技术偏好到实现要求的映射
- **强制执行技术细节提取检查清单**（≥80%完成率）
- **遵循严格的质量评分标准**（≥7/10分通过门控）
- **确保所有技术引用的准确性**（正确的分片文档引用格式）
- **完成Story质量验证流程**（8步骤验证过程）

### 对Dev Agent的一致性要求

- 严格按照story中的技术偏好摘要实现
- 在技术选择中优先考虑用户偏好
- 遇到技术偏好冲突时及时寻求澄清

### 对QA Agent的一致性要求

- 验证实现与原始技术偏好的合规性
- 检查技术栈使用的正确性
- 确保技术决策追溯链条的完整性

## 技术偏好一致性失效预防

### 常见失效模式识别

1. **传递断层**：架构文档未充分体现用户偏好
2. **理解偏差**：SM agent误解架构文档中的技术要求
3. **实现偏离**：Dev agent基于便利性偏离技术偏好
4. **验证缺失**：QA agent未检查技术偏好合规性
5. **技术准确性失效**：Story中技术细节与架构文档不一致
6. **质量门控失效**：低质量story进入开发流程

### 预防机制

1. **强制性检查点**：每个阶段都有技术偏好一致性验证
2. **源文档引用**：所有技术决策都追溯到原始偏好
3. **多重验证**：不同角色从不同角度验证一致性
4. **明确职责**：每个agent都有明确的技术偏好执行责任
5. **量化质量门控**：建立严格的评分标准和通过阈值
6. **双层技术验证**：SM Agent自检+Architect Agent专业审核
7. **强制性检查清单**：确保技术细节提取的完整性和准确性

## 实施效果监控

### 一致性指标

- **技术偏好覆盖率**：story中技术偏好要求的完整性
- **执行合规率**：实现与原始偏好的匹配度
- **验证有效性**：QA发现的技术偏好违规情况
- **追溯完整性**：技术决策到原始偏好的可追溯性
- **SM Agent质量评分率**：Story创建质量评分≥7/10的比例
- **技术提取完成率**：技术细节提取检查清单≥80%完成的比例
- **Architect审核通过率**：技术准确性审核≥7/10分通过的比例
- **质量门控有效性**：被质量门控拦截的低质量story比例

### 持续优化机制

1. **定期评估**：技术偏好传递效果的周期性评估
2. **反馈收集**：各agent在执行过程中的问题反馈
3. **流程改进**：基于实际效果的传递机制优化
4. **文档更新**：根据实践经验更新检查标准

## 用户技术偏好执行保证

这套机制确保用户在technical-preferences.md中表达的技术偏好能够：

✅ **完整传递**：从架构师到开发者的每个环节都保持完整性
✅ **准确理解**：各个agent正确理解并执行技术偏好
✅ **一致实现**：最终代码实现与原始技术偏好高度一致
✅ **可追溯验证**：技术决策的来源和执行过程完全可追溯

通过这套完整的技术偏好一致性传递保证机制，Orchestrix框架能够确保用户的技术愿景在整个开发流程中得到忠实执行，避免技术偏好在传递过程中的失真或遗漏。
