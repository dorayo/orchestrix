# Orchestrix Brownfield 开发指南

## 什么是 Brownfield 开发？

Brownfield 开发是指在现有项目基础上进行功能扩展、问题修复或系统现代化的开发模式。与全新项目（Greenfield）不同， Brownfield 开发需要理解现有代码、遵循约束条件，确保新变更无缝集成且不破坏现有功能。

## 何时使用 Brownfield 开发流程

### ✅ 适用场景

- 为现有应用添加新功能
- 修复复杂系统问题
- 集成新技术或服务
- 现代化遗留代码
- 大型代码库的局部改进

### ❌ 不适用场景

- 刚完成MVP的后续开发（建议直接扩展PRD）
- 简单的样式调整或配置修改
- 独立的新模块开发

## Brownfield 开发标准流程

Brownfield 开发遵循标准10步工作流，但在每个阶段加入特殊的现有系统考虑：

```
现有系统分析 → 标准10步流程（带 Brownfield 适配）→ 安全集成部署
```

---

## 第一阶段：现有系统分析

### 目标

全面理解现有系统，为后续规划提供准确上下文。

### 执行步骤

**1. 系统文档化**

```bash
# 推荐：使用Gemini Web（1M+令牌上下文）
# 上传完整项目或GitHub URL
@analyst
*document-project
```

**分析师任务：**

- 生成完整的系统架构文档
- 识别关键组件和依赖关系
- 记录现有技术栈和模式
- 标注潜在风险区域

**输出：** `existing-system-analysis.md`

**2. 影响范围评估**

- 确定变更影响的模块
- 识别集成点和依赖关系
- 评估技术债务和约束

---

## 第二阶段：需求与规划

### 基于项目复杂度选择深度

| 项目规模         | 策略选择    | 文档深度     |
| ---------------- | ----------- | ------------ |
| **大型复杂项目** | 完整PRD流程 | 详细需求文档 |
| **中等功能扩展** | 史诗级规划  | 功能级规划   |
| **小型修复**     | 故事级规划  | 任务级描述   |

### 执行步骤

**大型项目 - 完整PRD流程**

```bash
@pm
*create-doc brownfield-prd
```

**中等项目 - 史诗级规划**

```bash
@pm
*brownfield-create-epic
```

**小型项目 - 故事级规划**

```bash
@pm
*brownfield-create-story
```

### Brownfield 特殊考虑

- **兼容性要求**：确保不破坏现有功能
- **迁移策略**：定义数据和代码迁移方案
- **回滚计划**：制定失败恢复策略
- **性能影响**：评估对现有系统的性能影响

---

## 第三阶段：架构设计

### 目标

设计与现有系统无缝集成的技术架构。

### 执行步骤

```bash
@architect
*create-doc brownfield-architecture
```

### 架构师特殊职责

- **集成设计**：确保新组件与现有架构协调
- **模式一致**：遵循现有的编码和架构模式
- **风险缓解**：识别并解决技术风险
- **渐进策略**：设计分阶段实施方案

**输出：** `brownfield-architecture.md`

---

## 第四阶段：质量保证

### 目标

确保所有规划文档与现有系统保持一致。

### 执行步骤

```bash
@po
*execute-checklist po-master-checklist
```

### PO验证要点

- ✅ 需求与现有系统兼容
- ✅ 架构设计不引入冲突
- ✅ 集成方案可行且安全
- ✅ 风险缓解措施充分

---

## 第五阶段：开发执行

### 遵循标准开发流程

按照标准10步流程的后续步骤执行，但加入 Brownfield 特殊考虑：

**SM创建故事时：**

- 包含现有代码集成任务
- 添加兼容性测试要求
- 规划渐进式部署

**Dev实施时：**

- 遵循现有代码风格
- 优先保证向后兼容
- 实施功能开关

**QA审查时：**

- 重点测试集成点
- 执行回归测试
- 验证性能影响

---

## 最佳实践

### 🎯 核心原则

1. **安全第一**：永远不破坏现有功能
2. **渐进改进**：分阶段实施变更
3. **文档驱动**：详细记录所有变更
4. **测试保障**：完整的回归测试

### 💡 实施技巧

- **功能开关**：使用特性标志控制新功能
- **并行开发**：新旧系统暂时并存
- **数据迁移**：制定详细的数据迁移计划
- **监控告警**：实时监控系统健康状态

### ⚠️ 风险控制

- **备份策略**：实施前完整备份
- **回滚预案**：准备快速回滚方案
- **分批部署**：采用蓝绿部署或灰度发布
- **性能监控**：密切关注系统性能指标

## 快速决策指南

### 选择执行深度

```
项目需要多少开发时间？
├─ 1天以内 → 使用 brownfield-create-story
├─ 1-5天 → 使用 brownfield-create-epic
└─ 5天以上 → 使用完整PRD流程

变更影响多少个系统模块？
├─ 单一模块 → 简化流程
├─ 2-3个模块 → 中等深度
└─ 多个系统 → 完整流程
```

### 文档化深度选择

```
对现有系统熟悉程度？
├─ 非常熟悉 → 重点文档化变更区域
├─ 基本了解 → 文档化相关模块
└─ 不太熟悉 → 全面系统文档化
```

## 常见场景处理

### 场景1：添加新功能模块

1. **系统分析** → 识别集成点
2. **PRD规划** → 定义功能边界
3. **架构设计** → 设计集成方案
4. **渐进开发** → 功能开关控制

### 场景2：修复复杂Bug

1. **问题分析** → 定位根本原因
2. **故事规划** → 使用 brownfield-create-story
3. **回归测试** → 重点验证修复效果
4. **监控部署** → 密切观察系统表现

### 场景3：技术栈升级

1. **完整分析** → 全面评估影响
2. **分阶段计划** → 制定迁移路线图
3. **并行运行** → 新旧系统共存
4. **逐步切换** → 渐进式迁移

## 故障排除

### 问题：AI不理解现有代码库

**解决方案：**

- 使用Gemini Web上传完整项目
- 提供关键文件路径给document-project
- 补充业务逻辑说明

### 问题：生成的方案与现有模式冲突

**解决方案：**

- 在PRD创建时明确现有约束
- 向架构师提供详细的技术栈信息
- 要求方案遵循现有模式

### 问题：集成点不明确

**解决方案：**

- 重新运行系统分析，重点关注API和数据流
- 在架构设计阶段强调集成需求
- 请架构师详细设计接口方案
