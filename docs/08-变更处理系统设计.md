# 变更处理系统设计

> 版本: 1.0
> 状态: 设计稿
> 日期: 2025-01-18

## 1. 设计目标

### 1.1 问题陈述

原有的变更处理系统存在以下问题：

1. **入口过多**：4层变更处理（SM → PO → Architect → PM），导致流程复杂
2. **升级逻辑复杂**：多个决策文件处理升级链，难以理解和维护
3. **职责不清晰**：PO 同时承担路由和偏航处理，角色模糊
4. **缺乏统一的提案机制**：变更信息通过 HANDOFF 消息传递，无法追溯

### 1.2 设计原则

1. **入口明确**：只有两个变更维度 —— 技术变更和产品变更
2. **职责单一**：每个角色只做一件事
3. **提案驱动**：所有变更通过持久化的提案文件传递
4. **可追溯**：完整的变更历史记录

## 2. 系统架构

### 2.1 核心角色分工

| 角色          | 职责         | 输入             | 输出                    |
| ------------- | ------------ | ---------------- | ----------------------- |
| **Architect** | 处理技术变更 | 技术变更请求     | 技术提案 (TCP)          |
| **PM**        | 处理产品变更 | 产品变更请求     | 产品提案 (PCP)          |
| **PO**        | 智能路由分发 | 不确定的变更请求 | HANDOFF to Architect/PM |
| **SM**        | 提案落地执行 | 提案文件         | Story 创建/更新         |

### 2.2 简化后的流程

```
                    变更请求
                        │
        ┌───────────────┼───────────────┐
        │               │               │
        ▼               ▼               ▼
   明确技术变更    不确定/混合      明确产品变更
        │               │               │
        ▼               ▼               ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │Architect│    │   PO    │    │   PM    │
   └────┬────┘    └────┬────┘    └────┬────┘
        │              │              │
        │         分析并分发          │
        │         ┌────┴────┐        │
        │         │         │        │
        ▼         ▼         ▼        ▼
   ┌─────────────────────────────────────┐
   │           提案文件                   │
   │   (Technical / Product / 混合)      │
   └──────────────────┬──────────────────┘
                      │
                      ▼
              ┌──────────────┐
              │      SM      │
              │ apply-proposal│
              └──────┬───────┘
                     │
                     ▼
            创建/更新 Story
                     │
                     ▼
            标准开发流程
         SM → Architect → Dev → QA
```

### 2.3 与原流程对比

| 方面         | 原设计                 | 新设计                |
| ------------ | ---------------------- | --------------------- |
| **变更入口** | 4个（SM/PO/Arch/PM）   | 2个（Architect + PM） |
| **路由机制** | 复杂升级决策链         | PO 简单分发           |
| **PO 角色**  | 处理 Epic 级偏航       | 仅做智能路由          |
| **SM 角色**  | 处理偏航 + 创建Story   | 仅执行提案落地        |
| **信息传递** | HANDOFF 消息（易丢失） | 提案文件（可追溯）    |
| **决策文件** | 5+ 个升级决策文件      | 无需升级决策          |

## 3. 提案系统设计

### 3.1 提案类型

| 类型     | 前缀 | 全称                      | 处理者    |
| -------- | ---- | ------------------------- | --------- |
| 产品提案 | PCP  | Product Change Proposal   | PM        |
| 技术提案 | TCP  | Technical Change Proposal | Architect |

### 3.2 提案编号规则

#### 格式

```
{TYPE}-{YEAR}-{SEQUENCE}

TYPE:     PCP | TCP
YEAR:     4位年份
SEQUENCE: 3位自增序列，每年重置
```

#### 示例

```
PCP-2025-001  # 2025年第1个产品提案
PCP-2025-002  # 2025年第2个产品提案
TCP-2025-001  # 2025年第1个技术提案
TCP-2025-015  # 2025年第15个技术提案
```

#### 自增算法

```markdown
1. 扫描对应目录下的提案文件
2. 筛选当前年份的提案
3. 提取最大序列号
4. 新编号 = 最大序列号 + 1
5. 如果当前年份无提案，从 001 开始
```

### 3.3 提案状态

```
draft ──────────→ applied ──────────→ (完成)
  │
  │ (被新提案取代)
  ▼
superseded
```

| 状态         | 含义   | 触发条件                |
| ------------ | ------ | ----------------------- |
| `draft`      | 待处理 | 提案创建时              |
| `applied`    | 已应用 | SM 完成 Story 创建/更新 |
| `superseded` | 已废弃 | 被新提案取代            |

### 3.4 文件存储结构

```
docs/
└── proposals/
    ├── product/
    │   ├── PCP-2025-001-mvp-scope-reduction.md
    │   ├── PCP-2025-002-new-auth-feature.md
    │   └── ...
    └── tech/
        ├── TCP-2025-001-api-redesign.md
        ├── TCP-2025-002-db-migration.md
        └── ...
```

文件命名格式：`{PROPOSAL_ID}-{title-slug}.md`

```
## 5. 双向关联机制

### 5.1 关联场景

当一个变更同时涉及产品和技术时，需要创建两个关联的提案：

```

产品变更 + 技术变更（混合变更）
│
▼
PO 分析并分发
│
┌───────┴───────┐
▼ ▼
PM Architect
│ │
▼ ▼
PCP-2025-001 TCP-2025-001
│ │
└───────┬───────┘
│
双向关联

```

### 5.2 关联建立流程

```

Step 1: PM 创建产品提案
┌─────────────────────────────────────┐
│ PCP-2025-001 │
│ linkage: │
│ requires_tech_change: true │
│ related_tech_proposal: null ←────│── 待填充
└─────────────────────────────────────┘
│
▼
Step 2: HANDOFF to Architect
│
▼
Step 3: Architect 创建技术提案
┌─────────────────────────────────────┐
│ TCP-2025-001 │
│ linkage: │
│ related_product_proposal: ────────│──→ PCP-2025-001
└─────────────────────────────────────┘
│
▼
Step 4: Architect 回填产品提案
┌─────────────────────────────────────┐
│ PCP-2025-001 │
│ linkage: │
│ requires_tech_change: true │
│ related_tech_proposal: ───────────│──→ TCP-2025-001 (已填充)
└─────────────────────────────────────┘
│
▼
Step 5: 双向关联完成 → HANDOFF to SM

````

#### 产品提案中的关联字段
````

## 6. 任务文件设计

### 6.1 PO 路由任务 (po-route-change.md)

```markdown
# po-route-change.md

## 命令

\*route-change <change_description>

## 职责

分析变更请求，路由到正确的处理者。

## 输入

- change_description: 变更描述

## 处理流程

### Step 1: 分析变更类型

分析 change_description 中的关键词：

技术变更指示符:

- architecture, API, database, schema
- performance, security, infrastructure
- refactor, migration, integration

产品变更指示符:

- feature, requirement, user story
- MVP, scope, priority
- PRD, product, business

### Step 2: 判断并路由

IF 明确是技术变更:
→ 🎯 HANDOFF TO ARCHITECT: \*resolve-change
Context: {change_description}

ELSE IF 明确是产品变更:
→ 🎯 HANDOFF TO PM: \*revise-prd
Context: {change_description}

ELSE IF 混合变更（同时包含技术和产品指示符）:
→ 产品优先原则
→ 🎯 HANDOFF TO PM: \*revise-prd
Context: {change_description}
Note: 混合变更，产品优先处理，可能需要后续技术提案

ELSE (无法判断):
→ 询问用户澄清
→ "请问这个变更主要是产品需求变更还是技术架构变更？"

## 输出

HANDOFF 消息到 Architect 或 PM
```

### 6.2 SM 提案应用任务 (sm-apply-proposal.md)

```markdown
# sm-apply-proposal.md

## 命令

\*apply-proposal [proposal_id]

## 职责

读取提案文件，创建或更新对应的 Story。

## 输入

- proposal_id: (可选) 指定提案ID，如 PCP-2025-001 或 TCP-2025-001

## 处理流程

### Step 1: 确定待处理提案

IF proposal_id 已指定:
→ 定位并读取指定提案
→ 验证提案状态为 draft

ELSE (无参数):
→ 执行 Step 1a: 自动发现

#### Step 1a: 自动发现 draft 提案

1. 扫描产品提案目录: docs/proposals/product/
   - 筛选 status: draft
   - 按编号降序排列
   - 记录: latest_product_proposal

2. 扫描技术提案目录: docs/proposals/tech/
   - 筛选 status: draft
   - 按编号降序排列
   - 记录: latest_tech_proposal

3. 确定处理列表:

   IF 都没有 draft 提案:
   → 输出: "无待处理提案"
   → 结束

   IF 只有产品提案:
   → proposals_to_process = [latest_product_proposal]

   IF 只有技术提案:
   → proposals_to_process = [latest_tech_proposal]

   IF 两者都有:
   → 检查是否关联 (related\_\* 字段)

   IF 关联:
   → proposals_to_process = [
   latest_product_proposal, # 产品优先
   latest_tech_proposal
   ]

   IF 不关联:
   → 询问用户选择处理哪个
   → 或按创建时间顺序处理

### Step 2: 处理每个提案

FOR EACH proposal IN proposals_to_process:

1. 读取提案内容

2. 解析 Story Requirements 部分

3. FOR EACH requirement:

   IF action == "create":
   → 确定 Epic ID 和 Story ID
   → 基于 story-tmpl.yaml 生成完整 Story
   → 填充 AC（从 hints 扩展为完整 Given/When/Then）
   → 填充 Tasks（TDD 周期结构）
   → 执行 sm-story-quality 评分
   → 写入 Story 文件

   IF action == "modify":
   → 读取目标 Story 文件
   → 应用指定的修改
   → 更新 Story 版本信息
   → 重新评分

   IF action == "delete":
   → 标记 Story 状态为 Deprecated
   → 或移动到归档目录

4. 更新提案状态: draft → applied

5. 记录处理结果

### Step 3: 输出结果

1. 汇总创建/修改的 Story 列表

2. 输出 HANDOFF:

   🎯 HANDOFF TO ARCHITECT: \*review {first_story_id}

   Stories created/modified:
   - Story {id}: {title}
   - Story {id}: {title}
     ...

   Proposals applied:
   - {proposal_id}: {status}

## 处理流程图

                *apply-proposal [id?]
                        │
                   有指定 id?
                   ┌────┴────┐
                   ▼         ▼
                  YES        NO
                   │         │
                   ▼         ▼
             读取指定提案   扫描 draft 提案
                   │         │
                   │    ┌────┴────┐
                   │    ▼         ▼
                   │ product/   tech/
```

### A. 提案 ID 生成伪代码

```python
def generate_proposal_id(proposal_type: str) -> str:
    """
    生成提案编号

    Args:
        proposal_type: "product" 或 "technical"

    Returns:
        提案编号，如 "PCP-2025-001"
    """
    prefix = "PCP" if proposal_type == "product" else "TCP"
    year = datetime.now().year

    # 确定目录
    dir_path = f"docs/proposals/{'product' if proposal_type == 'product' else 'tech'}/"

    # 扫描现有提案
    pattern = f"{prefix}-{year}-*.md"
    existing = glob(f"{dir_path}/{pattern}")

    # 提取序列号
    sequences = []
    for file in existing:
        match = re.search(rf"{prefix}-{year}-(\d{{3}})", file)
        if match:
            sequences.append(int(match.group(1)))

    # 计算下一个序列号
    next_seq = max(sequences) + 1 if sequences else 1

    return f"{prefix}-{year}-{next_seq:03d}"
```

### B. 提案状态更新伪代码

```python
def update_proposal_status(proposal_path: str, new_status: str) -> None:
    """
    更新提案状态

    Args:
        proposal_path: 提案文件路径
        new_status: 新状态 (draft | applied | superseded)
    """
    content = read_file(proposal_path)

    # 解析 YAML front matter
    frontmatter, body = parse_frontmatter(content)

    # 更新状态
    frontmatter['metadata']['status'] = new_status

    # 写回文件
    write_file(proposal_path, format_frontmatter(frontmatter) + body)
```
