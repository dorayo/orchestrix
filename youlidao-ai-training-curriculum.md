# youlidao.ai 客户培训课程体系

> 面向非技术背景人员的 AI 驱动软件开发培训

---

## 培训设计理念

### 目标受众画像

- **背景**：企业主、产品经理、运营人员、创业者等非技术人员
- **痛点**：有软件需求但不懂技术，无法与开发团队有效沟通
- **期望**：能够使用 AI 工具独立完成简单软件项目，或更高效地管理外包开发

### 教学原则

1. **比喻先行**：用生活场景解释技术概念，建立心智模型
2. **实践为主**：70% 动手操作，30% 理论讲解
3. **成就驱动**：每个阶段都有可见的产出，保持学习动力
4. **检查清单**：提供模板和清单，降低认知负担

### 核心比喻体系：软件 = 数字餐厅

| 软件概念     | 餐厅比喻      | 通俗解释             |
| ------------ | ------------- | -------------------- |
| **前端**     | 餐厅前台      | 顾客看到和操作的界面 |
| **后端**     | 后厨          | 处理业务逻辑的地方   |
| **数据库**   | 仓库          | 存储所有数据的地方   |
| **API**      | 传菜口        | 前台和后厨沟通的窗口 |
| **Bug**      | 上错菜/做错菜 | 软件运行不符合预期   |
| **测试**     | 试吃          | 检验菜品是否合格     |
| **部署上线** | 开业迎客      | 软件正式对外使用     |

# 版本一：3 小时工作坊

> **目标**：快速上手，能独立完成一个简单项目
> **形式**：线下/线上工作坊
> **人数**：10-20 人

## 课程大纲

### 第一节：打开新世界的大门（40 分钟）

#### 1.1 破冰与期望对齐（10 分钟）

**互动问题**：

- "你有没有想过做一个 App 或网站，但因为不懂技术而放弃？"
- "你觉得做一个软件需要多少钱？多长时间？"

**本次培训目标**：

- 理解软件开发的基本流程
- 学会如何清晰表达需求
- 体验用 AI 完成一个小项目

#### 1.2 软件开发揭秘：餐厅比喻（15 分钟）

**核心内容**：

```
想象你要开一家餐厅...

1. 你需要想清楚：开什么餐厅？给谁吃？（需求分析）
2. 你需要设计：菜单长什么样？装修什么风格？（产品设计）
3. 你需要规划：厨房怎么布局？流程怎么走？（技术架构）
4. 你需要执行：装修、买设备、招人、培训（开发实现）
5. 你需要检验：试营业、调整菜品（测试验收）
6. 正式开业！（部署上线）

软件开发的流程，和开餐厅一模一样。
```

**视频/动画演示**：2 分钟动画展示软件开发全流程

#### 1.3 认识你的 AI 团队（15 分钟）

**类比引入**：

> "传统开发需要雇一个团队，现在你有 9 位 24 小时待命的 AI 员工"

**团队介绍**（配合头像卡片）：

| 角色          | 餐厅比喻    | 找他干什么                     |
| ------------- | ----------- | ------------------------------ |
| **Analyst**   | 市场调研员  | "帮我分析一下这个想法靠不靠谱" |
| **PM**        | 餐厅经理    | "帮我把想法变成详细的计划"     |
| **UX-Expert** | 室内设计师  | "帮我设计好看又好用的界面"     |
| **Architect** | 建筑工程师  | "帮我规划技术方案"             |
| **SM**        | 项目主管    | "帮我把任务拆成小块"           |
| **Dev**       | 厨师/施工队 | "帮我把设计变成真实的软件"     |

**关键认知**：

### 第二节：学会表达你的需求（20 分钟）

#### 2.1 为什么需求表达很重要（5 分钟）

**案例展示**：

```
❌ 模糊的需求：
"我想做一个电商网站"

❓ AI 的困惑：
- 卖什么？服装？食品？电子产品？
- 给谁用？C端消费者？B端批发商？
- 需要什么功能？商品展示？在线支付？会员系统？
- 什么风格？简约？奢华？可爱？

✅ 清晰的需求：
"我想做一个卖手工蛋糕的小程序，
目标用户是 25-40 岁的都市女性，
需要商品展示、在线下单、微信支付、
配送地址管理，风格要温馨治愈"
```

#### 10:30-10:45 ☕ 茶歇

#### 10:45-12:00 软件开发核心概念

**用餐厅比喻系统讲解**：

**第一部分：软件的组成（30 分钟）**

```
一个完整的「数字餐厅」包括：

🏠 前端（Front-end）= 餐厅前台
   - 顾客走进来看到的一切
   - 菜单、装修、服务员
   - 重点：好不好看、好不好用

🍳 后端（Back-end）= 后厨
   - 顾客看不到的地方
   - 接单、做菜、出餐的流程
   - 重点：逻辑对不对、效率高不高

🗄️ 数据库（Database）= 仓库
   - 存放所有食材（数据）
   - 记录所有订单（交易记录）
   - 重点：数据安全、查找方便

🚪 API = 传菜口
   - 前台和后厨沟通的标准窗口
   - "一号桌，红烧肉一份，少盐"
   - 重点：信息传递准确
```

**第二部分：开发流程（30 分钟）**

```
开餐厅的完整流程：

1️⃣ 市场调研（Analyst 阶段）
   - 这个位置适合开什么店？
   - 周围有什么竞争对手？
   - 目标客群是谁？

2️⃣ 商业规划（PM 阶段）
   - 餐厅定位是什么？
   - 菜单有哪些菜？
   - 价格策略如何？

3️⃣ 设计装修（UX 阶段）
   - 装修风格如何？
   - 动线怎么设计？
   - 菜单怎么排版？

4️⃣ 技术规划（Architect 阶段）
   - 厨房设备选什么？
   - 点餐系统用哪个？
   - 供应链怎么管理？

5️⃣ 任务拆分（SM 阶段）
   - 把大项目拆成小任务
   - 排优先级
   - 分配工作

6️⃣ 施工执行（Dev 阶段）
   - 按图施工
   - 设备安装
   - 系统调试

7️⃣ 验收检查（QA 阶段）
   - 试运营
   - 发现问题
   - 修复完善

8️⃣ 正式开业（部署上线）
   - 对外营业
   - 持续运营
   - 收集反馈
```

**小练习**：

- 给学员一个 App 截图
- 让他们识别哪些是前端、推测后端在做什么

### 下午（13:30-17:30）：需求表达深度训练

#### 13:30-14:30 需求的本质

**核心观点**：

> "写好需求，项目就成功了一半"

**好需求的标准（SMART 原则）**：

| 原则           | 含义     | 示例                                 |
| -------------- | -------- | ------------------------------------ |
| **S**pecific   | 具体明确 | ❌ "要快" ✅ "加载时间<3秒"          |
| **M**easurable | 可衡量   | ❌ "好看" ✅ "主色调 #4A148C"        |
| **A**chievable | 可实现   | ❌ "像淘宝一样" ✅ "商品列表+购物车" |
| **R**elevant   | 相关的   | 每个需求都要回答"为什么需要"         |
| **T**ime-bound | 有期限   | 明确优先级和阶段                     |

#### 14:30-15:30 需求文档实战

**需求文档模板详解**：

```markdown
# [项目名称] 需求文档

## 1. 项目背景

### 1.1 为什么要做这个项目？

[描述业务痛点或机会]

### 1.2 目标用户是谁？

[用户画像：年龄、职业、场景]

### 1.3 期望达成什么目标？

[量化的业务目标]

---

## 2. 功能需求

### 2.1 核心功能（MVP）

| 功能名称 | 用户故事                                   | 验收标准     |
| -------- | ------------------------------------------ | ------------ |
| [功能1]  | 作为[角色]，我想要[做什么]，以便[达成什么] | [怎样算完成] |

### 2.2 次要功能（Phase 2）

[列表形式]

### 2.3 暂不考虑的功能

[明确排除的范围]

---

## 3. 非功能需求

### 3.1 性能要求

- 页面加载时间：< X 秒
- 并发用户数：X 人

### 3.2 兼容性要求

- 支持的浏览器/设备

### 3.3 安全要求

- 数据加密、权限控制等

---

## 4. 设计参考

### 4.1 竞品参考

[截图 + 喜欢/不喜欢的点]

### 4.2 风格偏好

[色调、风格关键词]

## 5. 其他说明

[任何补充信息]
```

**分组练习**：

#### 15:30-15:45 ☕ 茶歇

#### 15:45-16:45 用户故事写作训练

**用户故事格式**：

```
作为 [某类用户]，
我想要 [完成某个操作]，
以便 [达成某个目的]。

验收标准：
- [ ] 条件1：当...时，应该...
- [ ] 条件2：当...时，应该...
```

**练习：把模糊需求变成用户故事**

| 模糊需求       | 转化后的用户故事                                                     |
| -------------- | -------------------------------------------------------------------- |
| "要有登录功能" | 作为新用户，我想要用手机号注册账号，以便保存我的数据                 |
| "要能搜索"     | 作为买家，我想要按关键词搜索商品，以便快速找到想买的东西             |
| "要好看"       | 作为用户，我想要看到清晰的商品图片和简洁的布局，以便快速了解商品信息 |

**个人练习**：

- 发放 10 个模糊需求

### 下午（13:30-17:30）：从设计到开发

#### 13:30-14:30 与 UX-Expert 协作

**UX-Expert 的能力**：

- 界面设计
- 交互设计
- 设计规范
- 组件库

**实操演练**：

- 基于 PRD 请 UX 设计界面
- 学习如何描述设计需求
- 如何给出设计反馈

**设计反馈话术**：

```
✅ 具体指出问题："首页的标题太小了，希望更突出"
✅ 说明原因："因为用户第一眼要看到核心卖点"
✅ 给出参考："类似 [某App] 那种大标题风格"

❌ 避免模糊评价："不太好看"、"感觉不对"
```

#### 14:30-15:00 与 Architect 协作

**Architect 的能力**：

- 技术选型
- 架构设计
- 数据库设计
- API 设计

**科普讲解**：

- 为什么需要架构师？（餐厅比喻：建筑工程师 vs 装修工人）
- 什么情况需要关注架构？（项目规模、性能要求、安全要求）

**实操演练**：

- 让 Architect 评估项目的技术方案
- 理解他输出的架构文档（不需要完全看懂，知道大概在说什么）

#### 15:00-15:15 ☕ 茶歇

#### 15:15-16:30 与 Dev 协作

**Dev 的能力**：

- 代码实现
- 功能开发
- Bug 修复
- 代码优化

**实操演练**：

- 让 Dev 实现一个具体功能
- 观察开发过程
- 学习如何描述 Bug 和修改需求

**Bug 报告模板**：

```markdown
## 问题描述

[一句话说明问题]

## 重现步骤

1. 打开 [页面]
2. 点击 [按钮]
3. 输入 [内容]
4. 出现 [问题]

## 期望结果

[应该是什么样]

## 实际结果

[现在是什么样]

## 截图（如有）

[附上截图]
```

#### 16:30-17:30 与 QA 协作

**QA 的能力**：

- 代码审查
- 测试验证
- 质量把关
- 问题追踪

**了解测试的类型**：

- 功能测试：能不能用？
- 界面测试：好不好看？
- 性能测试：快不快？
- 安全测试：安不安全？

**实操演练**：

- 让 QA 对已完成的功能做测试
- 理解测试报告
- 基于测试结果提出修改

**今日作业**：

- 完成项目的核心功能开发
- 准备明天的项目展示
