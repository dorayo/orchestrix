# youlidao.ai 客户培训课程体系（讲授版）

> 面向非技术背景人员的 AI 驱动软件开发认知培训
> 版本：v2.0 - 以讲授为主，聚焦认知构建

---

## 设计理念变化

| 维度     | v1.0（实操版）  | v2.0（讲授版）    |
| -------- | --------------- | ----------------- |
| **重心** | 动手操作        | 认知构建          |
| **比例** | 30% 讲 + 70% 练 | 70% 讲 + 30% 演示 |
| **目标** | 会操作          | 懂原理、能判断    |
| **深度** | 知道怎么做      | 知道为什么这么做  |

### 核心教学策略

1. **故事化**：每个概念都用真实故事引入
2. **可视化**：大量图示、流程图、对比表
3. **渐进式**：从日常经验出发，逐步引入专业概念
4. **演示代替实操**：讲师演示 + 学员观看理解

### 第一章：为什么软件开发这么难？（40 分钟）

> **本章目标**：让学员深刻理解传统软件开发的痛点，为后续解决方案做铺垫

#### 1.1 开场故事：一个项目的悲剧（10 分钟）

**真实案例讲述**：

```
某餐饮连锁企业想做一个会员管理系统...

第 1 个月：找了 3 家外包公司比价
  → 报价从 15 万到 80 万不等，完全看不懂差在哪

第 2 个月：选了中间价位的公司，签合同
  → 合同里写的功能清单，自己都看不太懂

第 3 个月：开发进行中，每周开会
  → 会上说的话互相听不懂，"接口"、"并发"、"部署"...

第 4 个月：第一次看到东西
  → 完全不是想象中的样子，但说不清哪里不对

第 5 个月：改了又改
  → 每改一个地方，另一个地方就出问题

第 6 个月：勉强上线
  → 员工不会用，顾客投诉多，系统经常崩

第 7 个月：继续修修补补
  → 又花了 10 万，还是一堆问题

第 8 个月：放弃了
  → 回到用 Excel 管理会员...

最终：花了 50 万，耗时 8 个月，什么都没得到。
```

**问学员**：这个故事里，问题出在哪？

#### 1.2 软件开发的三大鸿沟（15 分钟）

**用图示讲解**：

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   业务方                    技术方                   │
│   👔                         👨‍💻                    │
│                                                     │
│   "我想要一个                "需求不清楚             │
│    好用的系统"                没法开发"              │
│                                                     │
│         ════════ 鸿沟 1：语言不通 ════════          │
│                                                     │
│   "我说的很清楚了"          "你说的我理解了         │
│                              但不是这个意思"         │
│                                                     │
│         ════════ 鸿沟 2：认知不同 ════════          │
│                                                     │
│   "改一下应该很简单吧"      "这要重构整个系统"       │
│                                                     │
│         ════════ 鸿沟 3：期望错位 ════════          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**逐个讲解**：

**鸿沟 1：语言不通**

- 业务方说："我想要这个按钮点了之后有个好看的效果"
- 技术方听到的："你想要什么效果？CSS 动画？JS 特效？Lottie？"
- 结果：双方都觉得对方"不专业"

**鸿沟 2：认知不同**

- 业务方看软件：一个完整的"东西"
- 技术方看软件：无数个组件、接口、数据流的组合
- 就像：业主看到的是"一套房子"，建筑师看到的是"结构、水电、暖通..."

**鸿沟 3：期望错位**

- 业务方觉得：电脑做的，应该很快
- 技术方现实：写代码只占 20%，剩下 80% 是沟通、测试、修复
- 改需求的代价：业务方觉得是改几个字，技术方觉得是重新盖房

#### 1.3 传统解决方案为什么不够好（15 分钟）

**对比表格**：

| 方案           | 优点     | 致命问题                              |
| -------------- | -------- | ------------------------------------- |
| **自建团队**   | 可控性强 | 成本极高（5人团队年薪200万+），管理难 |
| **外包开发**   | 不用管人 | 沟通成本高，质量参差不齐，甩锅严重    |
| **低代码平台** | 门槛低   | 天花板明显，复杂需求做不了            |
| **模板建站**   | 便宜快速 | 千篇一律，无法定制                    |
| **自己学编程** | 省钱     | 学习曲线陡峭，时间成本高              |

**关键洞察**：

> 这些方案都没有解决根本问题——**业务方和技术实现之间的鸿沟**

### 第二章：AI 软件公司如何解决问题（50 分钟）

> **本章目标**：理解 youlidao.ai 的核心价值和底层逻辑

#### 2.1 换一个思路：如果 AI 能懂业务语言？（10 分钟）

**思想实验**：

```
想象一下，如果有一个技术团队：
- 能听懂你说的每一句话
- 会主动问你没说清楚的地方
- 24 小时随时待命
- 不会请假、不会离职、不会甩锅
- 每一步工作都让你看得见
- 而且，不贵

这就是我们正在做的事情。
```

**核心理念**：

```
传统模式：业务方 → 翻译 → 技术方 → 开发 → 产品
                ↑
              (鸿沟在这里)

AI 模式：业务方 → AI 团队 → 产品
              ↑
           (AI 来弥合鸿沟)
```

#### 2.2 youlidao.ai 的本质：标准化流程的图形化封装（20 分钟）

**重点讲解**：

> youlidao.ai 不是一个简单的聊天机器人，它背后是一套**经过验证的软件开发标准流程**——Orchestrix。

**什么是 Orchestrix？**

```
Orchestrix = Orchestra（管弦乐团）+ Matrix（矩阵）

就像一个管弦乐团：
- 有指挥（流程引擎）
- 有不同声部（专业化 AI 角色）
- 有乐谱（标准化工作流）
- 最终演奏出和谐的音乐（高质量软件）

每个 AI 不是各干各的，而是按照严格的流程协作。
```

**用图示解释层次关系**：

```
┌─────────────────────────────────────────────────────┐
│                 你看到的界面                         │
│              youlidao.ai 建造中心                    │
│         ┌─────────────────────────────┐            │
│         │  三区工作台 / AI 对话界面    │            │
│         └─────────────────────────────┘            │
├─────────────────────────────────────────────────────┤
│                 背后的引擎                           │
│              Orchestrix 框架                        │
│  ┌──────────┬──────────┬──────────┬──────────┐     │
│  │ 标准流程 │ 质量门禁 │ 状态管理 │ 文档模板 │     │
│  └──────────┴──────────┴──────────┴──────────┘     │
├─────────────────────────────────────────────────────┤
│                 底层能力                            │
│              Claude AI 大模型                       │
│         （理解、推理、生成代码）                     │
└─────────────────────────────────────────────────────┘

你不需要知道底层怎么工作，
但你需要知道：这不是随便聊聊，而是有章法的。
```

**为什么这很重要？**

- **可预期**：每个阶段产出什么，是确定的
- **可追溯**：出了问题，能知道哪个环节出的
- **可复制**：做 10 个项目，质量是稳定的
- **可学习**：AI 遵循的是业界最佳实践

#### 2.3 九大 AI 角色的专业分工（20 分钟）

**用"公司组织架构"来类比**：

```
┌─────────────────────────────────────────────────────┐
│                   你（老板/客户）                     │
│                        👔                           │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│                   战略规划部                         │
│  ┌─────────────────────────────────────────────┐   │
│  │  🔍 Analyst    📋 PM    🎨 UX    🏗️ Arch   ✅ PO │
│  │  市场分析     产品规划   设计    架构      质量   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  职责：想清楚"做什么"和"怎么做"                      │
│  产出：调研报告、PRD、设计稿、架构文档               │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│                   工程实施部                         │
│  ┌─────────────────────────────────────────────┐   │
│  │  📊 SM        🔧 Arch-Eng   💻 Dev    🧪 QA   │
│  │  任务拆分     技术把关       开发      测试   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  职责：把规划变成实际可运行的软件                    │
│  产出：用户故事、代码、测试报告                      │
└─────────────────────────────────────────────────────┘
```

**为什么要分这么多角色？**

**用医院类比**：

```
生病了去医院，你会遇到：
- 导诊台：告诉你挂什么科 → 类似 Analyst
- 门诊医生：诊断病情，制定治疗方案 → 类似 PM
- 检验科：做各种检查 → 类似 UX（信息收集）
- 专家会诊：复杂情况多方讨论 → 类似 Architect
- 护士长：安排护士执行 → 类似 SM
- 护士：打针、换药、执行医嘱 → 类似 Dev
- 质控科：检查医疗质量 → 类似 QA

为什么医院要分这么细？
因为专业的事要专业的人做，流程要标准化才能保证质量。

软件开发也是一样的道理。
```

**每个角色的具体职责（表格）**：

| 角色                   | 核心职责             | 什么时候找他             | 典型产出            |
| ---------------------- | -------------------- | ------------------------ | ------------------- |
| **Analyst**            | 市场分析、可行性评估 | 有想法但不确定靠不靠谱   | 调研报告、竞品分析  |
| **PM**                 | 需求梳理、功能规划   | 想法确定了，要细化成方案 | PRD（产品需求文档） |
| **UX-Expert**          | 界面设计、交互设计   | 需要设计怎么呈现给用户   | 设计稿、交互规范    |
| **Architect-Planning** | 技术选型、架构设计   | 需要规划技术方案         | 架构文档            |
| **PO**                 | 质量把关、一致性检查 | 检查各文档是否对齐       | 验收标准            |
| **SM**                 | 任务拆分、迭代管理   | 把大需求拆成小任务       | 用户故事            |
| **Architect-Eng**      | 技术评审、代码规范   | 确保代码符合架构要求     | 评审意见            |
| **Dev**                | 代码实现             | 把设计变成代码           | 可运行的代码        |
| **QA**                 | 测试验证             | 检查代码质量             | 测试报告            |

**关键认知**：

> 你不需要记住所有角色，只需要知道：**从 PM 或 Analyst 开始，他们会引导你找到对的人**。

### 第三章：标准化工作流详解（50 分钟）

> **本章目标**：理解软件开发的完整流程，知道每一步会发生什么

#### 3.1 为什么需要标准化流程？（10 分钟）

**反面案例**：

```
没有标准流程的开发：

老板："做一个商城"
开发：开始写代码...
（3天后）
开发："做好了，您看看"
老板："这不是我要的啊，购物车呢？支付呢？"
开发："您没说要啊..."
老板："这不是常识吗！"
开发：（内心）你怎么不早说...

循环往复，项目永远做不完。
```

**有标准流程的开发**：

```
第 1 步：Analyst 确认 - "您要做的是 B2C 电商还是 B2B？目标用户是谁？"
第 2 步：PM 细化 - "根据您的需求，功能清单如下，请确认"
第 3 步：UX 设计 - "界面是这个风格，流程是这样走，请确认"
第 4 步：Architect 规划 - "技术方案确定，准备开始开发"
第 5 步：SM 拆分 - "拆成 20 个小任务，按优先级排序"
第 6 步：Dev 开发 - 逐个完成，每完成一个可以看到
第 7 步：QA 测试 - 每个功能都经过测试才算完成

每一步都有确认，每一步都有产出，不会出现"不是我要的"这种情况。
```

#### 3.2 八步标准工作流（25 分钟）

**用流程图详细讲解**：

```
                    ┌─────────────────┐
                    │   Step 1        │
                    │   项目启动      │
                    │   Analyst       │
                    └────────┬────────┘
                             │
        产出：项目简报（Project Brief）
        内容：市场分析、可行性评估、风险提示
                             │
                             ▼
                    ┌─────────────────┐
                    │   Step 2        │
                    │   需求规划      │
                    │   PM            │
                    └────────┬────────┘
                             │
        产出：产品需求文档（PRD）
        内容：功能清单、用户流程、验收标准
                             │
                             ▼
                    ┌─────────────────┐
                    │   Step 3        │
                    │   设计规划      │
                    │   UX-Expert     │
                    └────────┬────────┘
                             │
        产出：前端设计规范（Front-end Spec）
        内容：界面设计、交互规范、组件定义
                             │
                             ▼
                    ┌─────────────────┐
                    │   Step 4        │
                    │   架构规划      │
                    │   Architect     │
                    └────────┬────────┘
                             │
        产出：架构设计文档
        内容：技术选型、系统结构、数据库设计、API设计
                             │
                             ▼
                    ┌─────────────────┐
                    │   Step 5        │
                    │   需求对齐      │
                    │   PM            │
                    └────────┬────────┘
                             │
        动作：根据架构限制更新 PRD
        确保：产品需求和技术方案对齐
                             │
                             ▼
                    ┌─────────────────┐
                    │   Step 6        │
                    │   质量验证      │
                    │   PO            │
                    └────────┬────────┘
                             │
        动作：检查所有文档的一致性
        产出：验证报告、Epic 拆分
                             │
                             ▼
                    ┌─────────────────┐
                    │   Step 7        │
                    │   任务拆分      │
                    │   SM            │
                    └────────┬────────┘
                             │
        产出：用户故事（User Stories）
        内容：具体任务、验收标准、优先级
                             │
                             ▼
                    ┌─────────────────┐
                    │   Step 8        │
                    │   开发迭代      │
                    │   Dev + QA      │
                    └─────────────────┘
                             │
        循环：开发 → 自测 → 代码审查 → 完成
        直到所有 Story 完成
```

**每一步详解**（讲师用案例演示）：

**Step 1-2：从模糊到清晰**

```
你说："我想做一个预约系统"

Analyst 会问：
- 预约什么？美容美发？医疗？餐厅？
- 谁来用？顾客？还是员工？
- 需要支付吗？
- 需要提醒吗？

PM 会输出：
┌─────────────────────────────────────────┐
│ 功能清单                                │
├─────────────────────────────────────────┤
│ ✅ 服务项目展示                         │
│ ✅ 可用时间查看                         │
│ ✅ 在线预约                             │
│ ✅ 预约管理（改期/取消）                │
│ ✅ 预约提醒（短信/微信）                │
│ ❌ 在线支付（Phase 2）                  │
│ ❌ 会员系统（Phase 2）                  │
└─────────────────────────────────────────┘

你能清楚看到：做什么、不做什么、以后做什么
```

**Step 3-4：从概念到设计**

```
UX 会输出：
- 每个页面长什么样
- 用户怎么一步步操作
- 按钮放在哪里、什么颜色

Architect 会输出：
- 用什么技术实现（你不需要懂，但知道有人在规划）
- 数据怎么存储
- 系统怎么扩展
```

**Step 5-6：确保对齐**

```
这一步非常重要：

有时候 UX 设计了一个很酷的效果，但技术上实现成本很高；
有时候 PM 写的需求和 UX 设计的界面对不上；
有时候架构方案和产品需求有冲突。

PO 的工作就是：
- 检查所有文档是否一致
- 发现问题及时纠正
- 确保大家说的是同一件事
```

**Step 7-8：从规划到执行**

```
SM 会把"做一个预约系统"拆成：

Story 1.1: 服务项目列表页
Story 1.2: 服务详情页
Story 2.1: 日历选择组件
Story 2.2: 时间段选择
Story 3.1: 预约确认流程
Story 3.2: 预约成功页
...

每个 Story：
- 有明确的验收标准
- 可以独立开发和测试
- 完成一个就能看到一个
```

#### 3.3 质量门禁系统（15 分钟）

**什么是质量门禁？**

```
门禁 = 不符合标准就不能通过

就像机场安检：
- 有违禁品 → 不能登机
- 证件不对 → 不能登机
- 不管你是谁，规则对所有人一样

软件开发的质量门禁：
- 代码有 Bug → 不能算完成
- 没有测试 → 不能算完成
- 不符合设计 → 不能算完成
```

**三道主要门禁**：

```
┌─────────────────────────────────────────────────────┐
│                  门禁 1：Story 质量门               │
│                  (SM 创建故事时)                    │
├─────────────────────────────────────────────────────┤
│ 检查项：                                            │
│ □ 故事描述是否清晰？                                │
│ □ 验收标准是否明确？                                │
│ □ 是否遗漏边界情况？                                │
│ □ 技术方案是否对齐？                                │
│                                                     │
│ 不通过 → 打回修改，不进入开发                       │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│                  门禁 2：开发完成门                 │
│                  (Dev 提交时)                       │
├─────────────────────────────────────────────────────┤
│ 检查项（7 项关键检查）：                            │
│ □ 所有测试通过？                                    │
│ □ 代码规范无错误？                                  │
│ □ 所有验收标准实现？                                │
│ □ 安全要求满足？                                    │
│ □ 无未处理异常？                                    │
│ □ 开发日志完整？                                    │
│                                                     │
│ 不通过 → 继续修改，不进入测试                       │
└─────────────────────────────────────────────────────┘
```

#### 15:15-16:15 Story 状态管理详解

**为什么要管理状态？**

```
传统项目管理的混乱：

老板："那个功能做完了吗？"
开发："做完了"
老板："那怎么还有 Bug？"
开发："功能做完了，Bug 是另一回事"
老板："？？？"

什么叫"做完"？没有统一标准，各说各话。
```

**8 种标准状态详解**：

```
Story 的生命周期：

┌─────────────────────────────────────────────────────────┐
│                        状态流转图                       │
└─────────────────────────────────────────────────────────┘

          ┌─────────┐
          │ Blocked │ ← 有问题，需要修改
          └────┬────┘
               │ (SM 修复)
               ▼
    ┌─────────────────────┐
    │ AwaitingArchReview  │ ← 等待架构师审查
    └──────────┬──────────┘
               │
      ┌────────┴────────┐
      │                 │
      ▼                 ▼
┌───────────┐    ┌───────────────┐
│ Approved  │    │RequiresRevision│ ← 需要修改
└─────┬─────┘    └───────┬───────┘
      │                  │
      │    (SM 修改后)   │
      │       ┌──────────┘
      ▼       ▼
┌─────────────────┐
│   InProgress    │ ← Dev 正在开发
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│     Review      │ ← QA 正在审查
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
┌───────┐  ┌───────────┐
│ Done  │  │ Escalated │ ← 问题升级，需要人工介入
└───────┘  └───────────┘
```

**状态含义速查表**：

| 状态                   | 含义           | 谁负责    | 下一步        |
| ---------------------- | -------------- | --------- | ------------- |
| **Blocked**            | 有问题无法继续 | SM        | 修复问题      |
| **AwaitingArchReview** | 等架构师看     | Architect | 审查通过/打回 |
| **RequiresRevision**   | 要改           | SM        | 修改后重审    |
| **Approved**           | 可以开发了     | Dev       | 开始写代码    |
| **InProgress**         | 正在开发       | Dev       | 开发完提交    |
| **Review**             | 正在审查       | QA        | 通过/打回     |
| **Done**               | 完成了         | -         | 🎉            |
| **Escalated**          | 搞不定         | 人类      | 人工介入      |

**关键机制：防止无限循环**

```
问题：如果 QA 一直不通过怎么办？

答案：有次数限制

架构师审查：最多 2 轮
如果 2 轮还不通过，升级处理

QA 审查：最多 3 轮
第 1 轮：正常标准
第 2 轮：稍微宽松
第 3 轮：只看关键问题
还不过：升级处理

这样避免了无限返工的死循环
```

**Step 5：需求对齐（30 分钟）**

**为什么需要这一步？**

```
常见问题：

设计师设计了一个很炫的动画效果...
架构师说：这个实现成本很高，要额外 2 周

产品写了"支持无限量用户"...
架构师说：这要用分布式架构，成本翻倍

设计师设计了一个实时聊天功能...
架构师说：这需要 WebSocket，原来的方案要改

如果不对齐，开发时才发现问题，代价巨大
```

**对齐的内容**：

```
PM 需要根据架构约束更新 PRD：

1. 功能调整
   - 有些功能技术上太复杂，简化或延后
   - 有些功能技术上有更好的方案

2. 性能预期调整
   - 根据架构能力设定合理预期
   - 比如：首屏加载 < 3 秒

3. 成本估算
   - 开发周期
   - 服务器成本
   - 第三方服务费用
```

**Step 7：SM 任务拆分（35 分钟）**

**SM 会做什么？**

```

   把 Epic 拆成 User Story

   Epic: 会员系统
```

**好的 Story vs 坏的 Story**：

```

"实现用户功能"

- 没有验收标准
- 范围不清楚

✅ 好的 Story：
"作为新用户，我想要用手机号注册账号，以便使用 App 的功能"

验收标准：
- [ ] 输入 11 位手机号，点击获取验证码
- [ ] 60 秒内收到短信验证码
- [ ] 输入正确验证码后注册成功
- [ ] 手机号已注册时提示"该手机号已注册"
- [ ] 验证码错误时提示"验证码错误"
```

**如何判断项目进度**：

```
总共 20 个 Story

真实进度：12/20 = 60%
```

```
3. 记录
```

```
   - 不是聊天机器人
   - 是经过验证的最佳实践

3. 你的角色是"指挥官"
   - 不需要懂技术细节
   - 但要知道想要什么、如何判断好坏

   - 不要期望一次完美
   - 先跑通，再优化
```
